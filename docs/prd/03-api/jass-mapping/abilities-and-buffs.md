## Canonical Go surface

```go
type Ability struct{ /* instance handle: unit × ability */ }
type AbilityType string
type Buff struct{ /* aura/buff instance */ }

func (u Unit) AddAbility(t AbilityType) Ability
func (u Unit) RemoveAbility(t AbilityType)
func (u Unit) Ability(t AbilityType) Ability      // zero-value no-op if absent (R-API-5)
func (u Unit) HasBuff(t BuffType) bool
func (u Unit) RemoveBuffs(pred BuffPredicate)     // RemoveBuffsBJ / UnitRemoveBuffsEx collapse

func (a Ability) Level() int
func (a Ability) SetLevel(n int)                  // also Inc/DecUnitAbilityLevel
func (a Ability) SetEnabled(b bool)               // UnitDisableAbility / BlzUnitDisableAbility
func (a Ability) SetPermanent(b bool)             // UnitMakeAbilityPermanent
func (a Ability) Cooldown() time.Duration
func (a Ability) StartCooldown(d time.Duration)
// D5 reflection collapse — one generic field accessor pair replaces 16 Blz natives:
func (a Ability) Field(f AbilityField) Value
func (a Ability) SetField(f AbilityField, v Value)

// Hero capability on Unit (heroes are units with a hero component):
func (u Unit) HeroLevel() int
func (u Unit) SetHeroLevel(n int, eyeCandy bool)
func (u Unit) AddXP(n int, eyeCandy bool)
func (u Unit) Strength(includeBonuses bool) int   // + Agility, Intelligence
func (u Unit) LearnSkill(t AbilityType)

// Tech/upgrades on Player:
func (p Player) SetTechResearched(t TechType, level int)
func (p Player) TechCount(t TechType, inProgress bool) int
func (p Player) SetTechAllowed(t TechType, max int)
```

## Dedup rules applied

| Rule | Application | Example |
|---|---|---|
| **D1** | passthrough/arg-swap BJs dropped | `UnitAddAbilityBJ`, `SetPlayerTechResearchedSwap` |
| **D2** | defaulted hero helpers collapse | `SetHeroLevelBJ` → `SetHeroLevel(n, eyeCandy)` |
| **D3** | `IncUnitAbilityLevel`/`DecUnitAbilityLevel`/`SetUnitAbilityLevel` → one setter | `Ability.SetLevel(a.Level()+1)` |
| **D4** | real logic kept once | hero-XP-table math (`bj_MELEE_HERO_LIMIT` style caps, tome handling) → data tables + `helpers` |
| **D5** | the 16-native `BlzGet/SetAbility{Boolean,Integer,Real,String}{,Level,LevelArray}Field` matrix → one typed `Field`/`SetField` pair | per-level field reflection with a typed `AbilityField` enum |

The Blz field-reflection matrix is the single biggest D5 win in the whole API:
~32 natives → 2 methods + a field-constant table generated by `jassgen` (R-AST-4).

## Subsystem dependencies

- **sim** (primary): ability instances, cooldown wheel (tick-quantized, R-EXEC-5), buff stacks with periodic effects, hero XP/attribute systems, tech tree gating train/build orders. Buffs come from pools (R-GC-2).
- **render**: cast animations (`Spell` clip per R-AST-3), buff visual attachments (delegated to [effects-and-graphics](effects-and-graphics.md)), command-card icons (UI).
- **asset**: ability/upgrade definitions are the largest `data/` tables (R-AST-1) — levels, targeting, cost, cooldown, field defaults. The WC3 object-data analogue lives here, not in code.

## Porting hazards

1. **WC3 abilities are data-driven engine behaviors**, not scripts — porting the *natives* is easy; the implicit ability *engine* (channel/cast-point/backswing phases, targeting validation, aura pulse cadence) is the real scope. The natives only make sense once that sim system exists (M3+).
2. **Ability vs buff vs tech ID confusion**: JASS uses raw integers for all three; LitD's distinct `AbilityType`/`BuffType`/`TechType` string keys prevent the classic four-cc mixups, but the manifest must classify every constant correctly.
3. **Field reflection writes mid-game** (`BlzSetAbility*Field`) mutate shared ability definitions vs per-instance values inconsistently in WC3. LitD rule: `SetField` is always per-instance copy-on-write; document divergence.
4. **`SelectHeroSkill` and skill-point economy** interact with UI (skill menu) — the sim must own point counts; UI only displays.
5. **Morph/metamorphosis abilities** change unit type in place — handle identity must survive (the `Unit` handle stays valid across morph), or every map script breaks.
6. **Cooldown determinism**: cooldown timers are gameplay state — tick-quantized fixed-point, included in the state hash (R-SIM-2).
