### 4c. The REAL blocker — the Blackwell sm_120 ORT stack (does not exist yet)
This is the crux. Findings (FSV'd):
- Calyx links ONNX Runtime as **`load-dynamic`** (`ort = "=2.0.0-rc.12"`, feature `api-24` → ONNX Runtime **1.26**). It loads `onnxruntime.dll` at runtime via `ORT_DYLIB_PATH`. Right now `ORT_DYLIB_PATH` (in `.env`) still points at the **Linux** path from aiwonder.
- `C:\code\Calyx-Dev\target\release\` has the CUDA **provider** DLLs (`onnxruntime_providers_cuda.dll` 92MB, tensorrt, shared) from a 6/15 build, **but `onnxruntime.dll` itself is MISSING** (confirmed just now: `Test-Path ...onnxruntime.dll` = False). The ORT build tree is gone.
- **No CUDA toolkit** (`nvcc` NOT on PATH; no `C:\Program Files\NVIDIA GPU Computing Toolkit\CUDA`). Driver is fine (610.x, CUDA UMD 13.3 per the systemspecs.md).
- **No Python** on PATH — and the **13 multimodal-adapter lenses run via a Python helper** (`multimodal_onnx_embed.py`) that needs `onnxruntime-gpu`. (The native onnx/fastembed lenses use the Rust `ort` crate.)
- Research (Exa) conclusion: **official ORT wheels do NOT ship sm_120 (Blackwell) kernels** → on stock ORT, `CUDAExecutionProvider` is unavailable and Calyx's fail-loud helper would (correctly) REFUSE rather than fall back to CPU. You need a **Blackwell sm_120 build**: either a prebuilt wheel (e.g. `Natfii/onnxruntime-gpu-blackwell` 1.24.1, CUDA13/cuDNN9) or build ORT 1.26 from source with `CMAKE_CUDA_ARCHITECTURES=120` to match Calyx's pinned 1.26.
