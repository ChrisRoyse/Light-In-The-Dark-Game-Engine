## 1. Vision statement

Light in the Dark (LitD) is a Warcraft III–inspired RTS game engine written in **pure Go**,
rendered through the [G3N](https://github.com/g3n/engine) OpenGL engine. Its public scripting
surface is a faithful, **deduplicated** Go port of the Warcraft III JASS API — every capability
of the original 2,521-function surface, expressed as the smallest idiomatic Go API that loses
no power.

The one-sentence pitch: *the WC3 map-making experience, rebuilt as a modern, deterministic
Go platform — without Blizzard's content, formats, or licensing baggage.* *(Revised
2026-06-11 per D-2026-06-11-21: the engine is **proprietary, permanently** — what is open is
the creator surface: the world archive format specification and the Lua scripting API
documentation are published; the engine source is not.)*

Three properties are non-negotiable and define the product:

1. **Deterministic by construction.** Identical inputs produce identical simulation states
   across runs and machines. No AI/ML at runtime, no proprietary tech, no wall-clock or
   goroutine-race inputs inside a tick. Determinism is the prerequisite for replays, lockstep
   multiplayer (committed milestone M7, D-2026-06-11-5), and CI-testable gameplay.
2. **Zero-cost stack.** Every runtime dependency is open source and **permissive-licensed**
   (BSD/MIT/Apache — copyleft is a hard exclusion, D-2026-06-11-21); every game model is
   CC0-licensed low-poly glTF. The asset budget for a complete WC3-style fantasy RTS is $0
   ([PRD §3.3](../../PRD.md#33-assets-cc0-low-poly-fantasy-rts-packs-zero-cost)).
3. **Low-tier hardware first.** The reference machine is a dual-core 2 GHz CPU with an Intel
   UHD 620 and 4 GB RAM. Performance budgets ([PRD §5.3](../../PRD.md#53-performance-budgets-acceptance-gates-low-tier-reference-machine-dual-core-2-ghz-intel-uhd-620-4-gb-ram))
   are acceptance gates enforced in CI, not aspirations.

## 2. What the engine is

- **An RTS-shaped engine.** Fixed 20 Hz simulation tick, 60 FPS interpolated rendering,
  locked RTS perspective camera, drag-select/control-group/smart-click input, grid
  pathfinding, fog of war, minimap. The shape of Warcraft III, generalized just enough to
  build any game in that genre family.
- **A Go library *and* a runtime-loadable creation platform.** *(Revised 2026-06-11 per
  D-2026-06-11-8/10/14.)* Systems and engine games are written in Go against `litd/api` and
  compiled. From M5, worlds are also authorable in **embedded deterministic Lua** — bindings
  generated from `api-manifest.json`, hard-sandboxed (no io/os/net, per-tick quotas,
  R-SEC-1) — and loadable at runtime from a single-file **world archive** (zip-based: map
  data + Lua + custom assets + manifest), no Go toolchain required
  ([PRD §5.6](../../PRD.md#56-moddingscripting)). An in-engine **World Editor** ships at M8;
  until then, GUI tooling is limited to the asset-validation CLI (see
  [Tooling](../09-roadmap/tooling.md)).
- **A WC3-grade AI scripting domain (M5.5).** The full `commonai` surface ships in v1
  (D-2026-06-11-6): a second sandboxed scheduler domain with isolated script contexts and
  command-stack messaging (R-EXEC-3) — the M6 skirmish opponent runs on it, not on a
  stopgap.
- **The engine behind a flagship game.** *(Added 2026-06-11 per D-2026-06-11-22/23/24.)*
  The M6 vertical slice is **v0.1 of the actual Light in the Dark game** — not a tech demo:
  art style, factions, and lore are established there and every subsequent milestone ships a
  better version of the same game; engine and game prove each other (D-24). Distribution is
  **own-site only** — downloads ship from the Light in the Dark Analytics site, no Steam, no
  itch.io, no GitHub releases (D-22). The hosted **world hub is committed M9** (D-23):
  static-friendly index, no account needed to download, co-hosting the M7 session relay.
- **A complete capability port of the JASS API.** Every native and BJ function in the source
  material is accounted for: mapped to exactly one canonical Go symbol or explicitly
  tombstoned with a reason. The audit report generated by `tools/jassgen` proves this
  mechanically ([PRD §4.2](../../PRD.md#42-deduplication-policy-the-complex-version-only-rule),
  [Tooling §2](../09-roadmap/tooling.md)).
- **Layered with one hard rule.** `litd/api` (public) sits over `litd/sim` (deterministic,
  headless ECS core), `litd/render` (G3N presentation), and `litd/asset` (GLB/atlas/audio
  pipeline). Simulation never imports render; render never mutates simulation
  ([PRD §4.1](../../PRD.md#41-architecture-two-layers-one-implementation)). The public API
  exposes zero G3N types (R-API-6), so the renderer is swappable in principle.

## 3. What the engine is not

These are deliberate exclusions, restated from
[PRD §2.2](../../PRD.md#22-non-goals-v1) and elaborated in
[Goals and Non-Goals](./goals-and-non-goals.md):

- **Not a Warcraft III remake.** No Blizzard content, campaigns, balance data, art, or file
  formats (MDX/MDL explicitly out). Only the *API shape* is ported — function signatures are
  facts about an interface; every implementation body, all data, and all assets are original
  or CC0.
- **Not a general-purpose engine.** It does not compete with Unity or Godot. Systems that an
  RTS does not need (general physics, first-person cameras, open-world streaming) are out of
  scope even where G3N could support them.
- **Not networked before M7.** *(Revised 2026-06-11 per D-2026-06-11-5.)* Multiplayer is a
  committed lockstep milestone (M7), no longer a v2 maybe; M0–M6 still ship no netcode, but
  determinism and the command stream are its load-bearing prerequisites.
- **Not mobile or console.** Desktop Windows/Linux/macOS only — G3N's supported platforms.
- **Not open-source, and not store-distributed.** *(Added 2026-06-11 per
  D-2026-06-11-21/22.)* The engine is proprietary permanently; the public surface is the
  world archive format spec and the Lua API docs. Distribution is from our own site only —
  no third-party stores or SDKs.

## 4. Target users

### 4.1 Go game developers

Developers who want to build an RTS (or RTS-adjacent game: MOBA, tower defense, hero
defense — the genres WC3 custom maps invented) in Go without assembling an engine from
scratch. For them, LitD offers:

- An idiomatic API: ~20 public noun types with methods (`Unit.SetLife`), value-type math
  (`Vec2`, `Angle`), options structs, Go closures as event handlers — no handle zoo, no
  manual `RemoveLocation` memory rituals.
- A headless, deterministic sim core they can unit-test in ordinary `go test` runs.
- A documented, CI-enforced performance contract so games inherit the low-tier baseline.

### 4.2 WC3 modders migrating to a real language

JASS and vJass authors with years of map-making knowledge whose mental model is the WC3 API.
For them, LitD offers:

- **Capability parity guaranteed by audit.** Anything expressible against the 1,536
  `common.j` natives and 985 `blizzard.j` BJ functions is expressible against `litd/api` —
  the manifest proves it function by function.
- **A generated JASS→Go mapping table** in the docs ([PRD §9.2](../../PRD.md#9-open-questions--decided-2026-06-11)):
  look up `SetUnitLifeBJ`, find `Unit.SetLife`. Migration is a lookup, not archaeology.
- **Familiar runtime semantics** where they matter: cooperative scheduling inside the sim
  tick, synchronous event handlers in deterministic order, tick-quantized waits — the JASS
  execution model formalized as R-EXEC-1…5
  ([PRD §4.4](../../PRD.md#44-execution-model-semantics-from-the-jass-runtime)).

A secondary audience — contributors to the engine itself — gets a vendored G3N fork,
data-oriented ECS internals, and a zero-allocation hot-path discipline (R-GC-1…5) worth
studying in its own right, but the product is designed for the two groups above.

## 5. Source material: the `components.json` / `blizzard.json` clarification

This deserves a precise statement because it shaped the project's data pipeline.

**The originally requested `components.json` and `blizzard.json` files do not exist** in the
[cipherxof/war3-types](https://github.com/cipherxof/war3-types) repository, and never did.
There is no upstream JSON manifest of the WC3 API to consume. What does exist — and what
[PRD §1.1](../../PRD.md#11-source-material-clarification) adopts as the API basis — is:

| File | Contents | Count |
|---|---|---|
| `repoes/war3-types/scripts/common.j` | Native (engine-level) JASS API | **1,536 natives** |
| `repoes/war3-types/scripts/blizzard.j` | Blizzard helper ("BJ") wrapper functions | **985 functions** |
| `repoes/war3-types/core/common.d.ts` | Typed version of common.j (1,534 declarations) | — |
| `repoes/war3-types/core/blizzard.d.ts` | Typed version of blizzard.j (985 declarations) | — |
| `repoes/war3-types/core/commonai.d.ts` | AI scripting natives (716 lines) | — |

Together the `.j` files define the **2,521-function** capability surface the port must
account for. The `.d.ts` files supply the type information the raw JASS declarations lack
(richer parameter typing, handle subtype hierarchy — 60+ handle subtypes).

**The JSON manifest is an output, not an input.** Where downstream tooling needs a
machine-readable manifest — and it does: code generation, the deduplication audit, the
migration mapping table — that file is `api-manifest.json`, **generated** by the
`tools/jassgen` parser from the five source files above (R-AST-4,
[PRD §6](../../PRD.md#6-asset--data-pipeline)). It is the de facto realization of the
requested "components/blizzard JSON": one entry per source function, its D1–D5
deduplication classification, and its canonical Go mapping. Its schema, generation
pipeline, and example entries are specified in [Tooling](../09-roadmap/tooling.md).

Practical consequences:

- Any document or ticket referring to `components.json` or `blizzard.json` should be read
  as referring to `api-manifest.json`.
- The manifest is regenerable at any time; it is build output, reviewed but never
  hand-edited (hand classification overrides live in a separate overrides file consumed by
  `jassgen` — see [Tooling §2.4](../09-roadmap/tooling.md)).
- M2 ([Milestones](../09-roadmap/milestones.md)) gates on the manifest covering all 2,521
  functions with a D1–D5 classification.

## 6. The components

A reader-level map of the system, expanding
[PRD §4.1](../../PRD.md#41-architecture-two-layers-one-implementation). *(Revised
2026-06-11: Lua scripting surface, AI domain, world archive, World Editor, and `tools/assetgen`
rows added per D-2026-06-11-6/8/10/12/14/15.)*

| Component | Role | Key constraints |
|---|---|---|
| `litd/api` | Public, idiomatic Go API — what game developers import | ~20 public types; zero G3N types in signatures (R-API-6); the only supported entry point |
| `litd/api/helpers` | BJ-derived helpers with real logic (D4 cases: `PolledWait`, melee-start utilities) | Layered strictly on top of core; never shadows canonical functions |
| `litd/sim` | Deterministic simulation core: ECS (struct-of-arrays), 20 Hz fixed tick, movement, pathfinding, combat, cooperative script scheduler | No rendering imports; headless-runnable (R-SIM-4); zero allocations per tick (R-GC-1); single seeded PRNG; no `map` iteration in gameplay code |
| `litd/render` | G3N presentation layer: reads sim state, interpolates, never writes | Draw-call budget ≤ 300/frame; 1 directional + ambient light in gameplay; custom passes for fog of war, minimap, selection circles |
| `litd/asset` | GLB/atlas/audio loading pipeline | Core glTF 2.0 only (`KHR_materials_unlit` excepted); `.ogg` audio; data tables in JSON/TOML under `data/` |
| `litd/script` (M5; package name provisional) | Embedded deterministic Lua VM — **vendored gopher-lua fork with four LITD patches** (D-2026-06-11-25: instruction budget, deterministic mathlib, coroutine persister, pooling + cross-arch CI test); bindings **generated from `api-manifest.json`** (D-2026-06-11-8) | Hard sandbox (R-SEC-1, D-2026-06-11-20): no io/os/net, game API only, per-tick instruction + memory quotas; worlds runtime-loadable |
| AI domain (M5.5; package name provisional, e.g. `litd/ai`) | Second sandboxed scheduler domain hosting the full `commonai` port (D-2026-06-11-6) | Isolated script contexts, no shared globals with map scripts; command-stack messaging only (R-EXEC-3); serializable like the map-script scheduler |
| World archive format (defined M6) | Single-file world container: map data + Lua + custom assets + manifest with content hashes and engine-version requirements (D-2026-06-11-14) | Zip-based, open, publicly documented; carries hosting metadata for the **committed M9 hub** (D-2026-06-11-23) from day one |
| World Editor (M8) | In-engine visual editor: terrain sculpt/paint, unit/doodad placement, map metadata, world-archive save; campaign menu/mission-flow UI (D-2026-06-11-10/15) | Built on the public API and the archive format; trigger-GUI authoring deferred — Lua covers logic |
| `tools/jassgen` | Parser/codegen: `.j` + `.d.ts` → `api-manifest.json` → API stubs + Lua bindings + audit report | Specified in [Tooling](../09-roadmap/tooling.md) |
| `tools/assetcheck` | Asset-validation CLI: core-glTF check, triangle/atlas budgets, required animation clips | Build-time gate (R-FMT-2, R-AST-2/3); specified in [Tooling](../09-roadmap/tooling.md) |
| `tools/assetgen` | Build-time generative pipeline for asset categories with no CC0 source: portraits, spell VFX textures, voice lines, UI icons, terrain splat/cliff sets (D-2026-06-11-12, R-AST-5) | Asset-build time only, hand-curated, provenance-recorded; zero runtime AI (G4.6); specified in [Tooling](../09-roadmap/tooling.md) |
| `repoes/engine` | Vendored G3N fork | We own maintenance and patches (instancing, glTF loader fixes) |
| `repoes/war3-types` | Vendored source material (read-only input to `jassgen`) | Never shipped; reference data only |

## 7. Why this combination wins

- **Low-poly true 3D, not 2.5D**, because WC3 itself was true 3D and its real failing was
  missing culling/LOD — not the dimensionality. A locked RTS camera makes 3D cheap on
  integrated GPUs, and the CC0 ecosystem is overwhelmingly low-poly glTF
  ([PRD §3.1](../../PRD.md#31-rendering-dimensionality-low-poly-3d-not-225d)).
- **glTF 2.0 binary (.glb), core profile**, because it is the fastest-loading,
  lowest-memory open format G3N supports, constrained to core profile to dodge G3N's
  partial extension support ([PRD §3.2](../../PRD.md#32-model-format-gltf-20-binary-glb-core-profile-only)).
- **Go with GC discipline**, because compiled Go gives zero-interpreter-cost scripting and
  trivially testable determinism, and the GC is a non-issue when steady-state allocation is
  held at zero by pooling and CI-enforced `AllocsPerRun` gates
  ([PRD §5.3.1](../../PRD.md#531-go-garbage-collection-discipline)).
- **Deduplication as a feature**, because collapsing 2,521 functions into a minimal
  canonical surface is what makes the API learnable — and the mechanical D1–D5 policy plus
  generated audit is what makes "minimal" provably lossless.
